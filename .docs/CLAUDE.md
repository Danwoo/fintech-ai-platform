# CLAUDE.md — `.docs` 작성 규칙

`.docs/` 는 `full-stack-template` 과 파생 서비스가 공유하는 **팀 기술 문서 집약소**다. 세 종류를 담는다 — ① 서비스를 가로지르는 **개념·흐름·기법**(`3-기법`/`4-아키텍처`), ② 신규 합류자 **개발 입문 가이드**(`2-개발가이드`), ③ 서버·CI·도구 **환경/운영 runbook**(`1-개발환경`/`5-인프라셋팅`/`6-도구`).

코드 패턴/룰의 SoT 는 repo [`.claude/docs/`](../.claude/docs/)이며, 서비스별 레이어 설명은 각 폴더 `CLAUDE.md`다. 겹치지 않게 쓰고, 입문 가이드는 깊은 룰을 그쪽으로 **링크**한다. 문서 목록·진입은 [README.md](README.md).

정본 예시 — 새 문서는 종류에 맞는 골격을 따른다:

- [4-아키텍처/saas-멀티테넌트.md](4-아키텍처/saas-멀티테넌트.md) — **흐름 중심**(설계 결정 + 표, 코드블록 없음)
- [3-기법/동시성-출처인덱스.md](3-기법/동시성-출처인덱스.md) — **코드 인덱스**(기법 + 출처앵커 코드블록 + 매트릭스)
- [3-기법/동시성-가이드.md](3-기법/동시성-가이드.md) — **가이드**(결정 규칙 + 예시 코드 템플릿, 출처앵커 없음)
- [2-개발가이드/fastapi-백엔드개발.md](2-개발가이드/fastapi-백엔드개발.md) — **입문**(스택 개념 설명 + `.claude/docs/`·CLAUDE.md 로 링크)
- [5-인프라셋팅/ubuntu-서버셋팅.md](5-인프라셋팅/ubuntu-서버셋팅.md) — **runbook**(절차 + `✅ 검증` 콜아웃)

## 렌더 타깃 (링크 규칙의 이유)

- **Git web** (정본 뷰): 폴더를 열면 그 안 `README.md` 가 자동 렌더 = 진입 페이지. 상대경로 링크 동작.
- **Obsidian**: `.docs` 만 통째로 빼서 vault 로 연다. 문서끼리 링크는 살고, 소스코드 링크는 vault 밖이라 안 열린다(읽기용 prose 라 무방).

## 폴더

폴더 이름의 **번호 = 읽는 순서**(`1`→`6`).

| 폴더 | 들어가는 것 | 종류 |
|---|---|---|
| `1-개발환경/` | 개발 시작 전 파이썬·패키지 준비 (uv) | runbook |
| `2-개발가이드/` | 개발 입문 가이드 (백엔드/프론트/컨테이너/MCP) — 깊은 룰은 `.claude/docs/` 로 링크 | 입문 |
| `3-기법/` | 코딩 기법 (동시성 · LLM 프롬프트/가드레일 · 출처 인덱스) | 기법 |
| `4-아키텍처/` | 시스템 설계·흐름 (MSA·SaaS·인증·RAG·MCP …) | 흐름 |
| `5-인프라셋팅/` | 서버·PC 셋업 (WSL2·Ubuntu·LLM서빙·Docker) | runbook |
| `6-도구/` | 개발·운영 도구 (컨텍스트 절약 도구, 외부서비스 모니터링 …) | 가이드/runbook |

주제가 어디에도 안 맞으면 폴더를 새 번호로 만들고, README 폴더 표와 이 표에 같이 한 줄 추가.

## 문서 골격

파일명 `<folder>/<주제>.md` (kebab-case, 날짜·작성자 접미사 금지). 공통 머리:

````md
# 제목 — 부제                       ← 부제는 다루는 범위/축을 나열
> <한 줄 요약: 무엇을 모았나. 범위(전 서비스 공통 / 특정 서비스 전용 / 특정 프로젝트 한정)를 이 줄에 못박는다>

---
## 0. …                             ← 큰 절 `## N.`, 하위 `### N.M`, 절 사이 `---`
(본문)
---
관련 문서: [A](경로) · [B](경로)     ← 다른 문서로의 cross-link 은 맨 끝 footer 한 줄로
````

- **첫 `>` 요약 1개만** 둔다. 범위 못박기를 그 요약에 흡수.
- **금지**: `> **읽는 법**` 블록 · 목차(ToC) 줄 · "대상 독자 / 선행 지식 / 학습 목표 / 읽고 나면 ~할 수 있다 / ~개요다 / ~온보딩 문서다" 류 강의식 framing. (무엇을 다루는지 알려주는 사실 문장은 OK — 금지는 독자·선행·학습목표 framing 과 "읽는 법" 라벨뿐.)
- 다른 문서로 가는 cross-link 은 맨 끝 `---` 아래 `관련 문서:` 한 줄(또는 짧은 목록)로 모은다. 본문에 자연스러운 인라인 링크가 이미 있으면 footer 중복은 불필요.
- 헤딩에 **이모지 금지**. 본문 헤딩 번호는 `N.` 형식.

**종류별 변형** (공통 머리는 동일, 본문 스타일만 다름):

| 종류 | 본문 | 코드블록 |
|---|---|---|
| 흐름 (4-아키텍처) | 설계 결정 + 표 + `>` 근거 콜아웃 | 보통 없음 |
| 코드 인덱스 (3-기법 실제적용) | 기법 + 매트릭스(부록 A) + do/don't(부록 B) | **출처 앵커 필수** (아래) |
| 가이드 (3-기법 표준·6-도구) | 결정 규칙 + 체크리스트 + DO/DON'T | 예시 코드 (출처 없이) |
| runbook (1-개발환경/5-인프라셋팅) | `## 단계` 절차, 명령 블록, 각 단계 끝 `✅ **검증**` 콜아웃 | 실행할 명령/설정 그대로 |
| 입문 (2-개발가이드) | 개념·how-to 설명 + 표/다이어그램. **사용 중인 스택 기준**, 깊은 룰·스캐폴드는 `.claude/docs/` + 해당 CLAUDE.md 로 링크 (중복 금지). 작성자·연도 머리말 금지 | 예시 코드 (출처 앵커 없음) |

## 코드 블록

- 언어 지정 펜스(`python` / `bash` / `typescript`).
- **코드 인덱스 문서만 출처 앵커 필수**: 첫 줄 `# <서비스-상대 경로>:<line>` (예: `# market-data-mcp-service/app/main.py:31`), 관련 없는 부분은 `...` 생략. 블록 뒤 정본 링크 + 반복 서비스는 경로만 나열.
- 가이드/교육/runbook 의 코드는 **설명용 예시·실행 명령**이라 출처 앵커를 달지 않는다 (없는 앵커를 지어내지 말 것).

## 표 · 블록인용

- 비교/매트릭스는 표로. `●`·`✅/❌` 그리드는 가운데 정렬(`:---:`) + 범례 한 줄.
- `>` 블록인용 = 결정 근거·주의·"왜 이렇게" 콜아웃. 본문이 "무엇"이면 `>` 가 "왜".
- runbook 의 `✅ **검증**` 은 단계가 성공했는지 확인하는 명령/기대출력 콜아웃. `⚠️` 는 되돌리기 힘든 주의.

## 링크

- **문서끼리**: 표준 마크다운 상대링크 `[제목](../4-아키텍처/saas-멀티테넌트.md)`. `[[위키링크]]` 금지 — Git web 에선 글자로 렌더된다. `.md` 확장자 유지.
- **소스코드**: `.docs/<folder>/` 기준 상대경로 + 라인앵커 — repo 루트까지 `../../`, 타 서비스(`01-services/`)까지 `../../../`, 줄은 `#Lxx`.
- **DRY**: 한 패턴은 대표 예시 1개만 정본 링크, 반복 서비스는 경로만. 같은 링크 도배 금지. 같은 사실을 두 문서에 복붙하지 말고 한쪽을 정본으로 두고 **링크**한다 (교육자료가 정본과 어긋나면 `> 참고` 콜아웃으로 정본을 가리킨다).

## 시크릿 정책

`.docs` 는 git 에 push + Obsidian vault 로 공유된다 → **평문 시크릿(키·토큰·비밀번호)을 두지 않는다.** 키가 필요한 문서는 `<NAVER_CLIENT_SECRET>` 식 플레이스홀더로 쓰고, "실제 값은 properties 파일/비밀번호 매니저 참조" 라고 한 줄 단다. 공개돼도 무방한 값(클라이언트 노출 GA ID, 공개 도메인)은 예외.

## 톤

팀 위키. 종류별로 톤을 달리한다.

- **입문 (2-개발가이드)** — 처음 읽어도 이해되게. 핵심은 "정보 나열"이 아니라 "이해시키기" — 상세 방법론과 체크리스트는 아래 **[## 교육 문서 작성법](#교육-문서-작성법-입문-가이드용)**.
- **레퍼런스 (4-아키텍처/3-기법)** — terse. 흐름·결정 근거 중심, 구현 디테일 나열·보고서/마이그레이션 서사 금지.
- 공통 — **과도한 서술 지양**: 항목 나열은 개조식(불릿·짧은 절)으로 스캔 가능하게. 흐름·토폴로지·순차 절차·결정 분기는 **mermaid flowchart 로** 보여준다(이미 그려졌으면 중복 금지, 억지 금지). 수치·위치는 코드 기준 사실로(`~74 파일`, `Semaphore(2)`, `#Lxx`) — 추정 금지. 헤딩 이모지 금지.

## 교육 문서 작성법 (입문 가이드용)

`2-개발가이드/` 같은 입문 문서는 **신입을 이해시키는 것**이 목표다. 정보를 다 적었다고 좋은 교육자료가 아니다 — 아래는 "교육하기 좋은 문서"의 작성법이자, **기존 문서를 리팩토링할 때의 기준**이다.

### 입문 문서 권장 흐름

```text
# 제목 — 부제
> 한 줄 요약 (+범위 못박기)

## (큰 그림)   ← 한 줄 또는 mermaid 다이어그램으로 전체부터. 숲을 보고 나무로.
## 핵심 개념    ← 무엇·왜 (동기 먼저, 메커니즘 나중)
## 동작 예시    ← 어떻게: 동작 코드 + before/after, 가능하면 end-to-end 한 개
## 직접 해보기  ← scaffold 호출 / runbook ✅검증 / 스스로 점검 질문
## 흔한 실수    ← 초보자가 빠지는 함정 (anti-pattern 으로 링크)
## 요약/체크리스트
```

모든 절을 다 둘 필요는 없다 — 주제에 맞게 취사. 단 **큰 그림 → 개념(왜) → 예시(어떻게)** 순서는 지킨다.

### 8가지 원칙 (작성·리팩토링 체크리스트)

1. **무엇 → 왜 → 어떻게** — 메커니즘 전에 동기·개념. "이게 뭔지, 왜 필요한지" 먼저, 그 다음 사용법.
2. **큰 그림 먼저** — 세부 전에 전체 흐름(한 줄/다이어그램)을 줘서 길을 잃지 않게.
3. **점진적 공개** — 흔한 80% 케이스를 먼저 완결하고, 엣지·예외·고급은 뒤나 부록으로. 첫 화면에 복잡도를 쏟지 않는다.
4. **구체 예시 필수** — 추상 규칙 뒤엔 항상 **동작하는 코드 + `❌ before` / `✅ after`**. 가능하면 한 기능을 처음부터 끝까지 따라가는 **worked example** 하나.
5. **용어 풀이** — 약어·전문용어는 처음 등장할 때 한 줄로 푼다(예: "DI(의존성 주입) = 객체를 직접 만들지 않고 밖에서 받아 쓰기"). 빗대기(analogy)도 적극.
6. **왜·실패 모드 콜아웃** — `>` 로 결정 근거와 "이렇게 안 하면 무슨 일이 나는지"를 보여준다. 실패를 보여주면 규칙이 기억된다.
7. **시각적으로 스캔 가능** — 짧은 단락 + 표 + 흐름도 + 리스트. 긴 산문 벽 금지.
8. **연결하되 중복 금지** — 깊은 룰/전체 스캐폴드는 `.claude/docs`·CLAUDE.md·다른 .docs 로 **링크**(이해에 필요한 핵심 예시만 인라인). 같은 내용 복붙 금지.

### 기존 문서 → 교육 문서 리팩토링 절차

대상 문서를 위 8원칙으로 진단하고 고친다:

- **보고서·나열형** → 맨 앞에 큰 그림을 얹고 "왜"를 앞으로 끌어온다. 길게 늘어진 서술은 개조식으로. 변경 이력/마이그레이션 서사는 삭제.
- **규칙만 있고 예시 없음** → before/after·worked example 삽입.
- **용어 폭탄** → 첫 등장 정의 + 필요 시 analogy.
- **한 문서에 주제 과다** → 먼저 **부록·하위 절·점진적 공개**로 *한 문서 안에서* 해소. 별 파일 분리는 **마지막 수단** — "각자 따로 읽어도 완결되는, 서로 독립한 reader-task 둘"일 때만. 한 주제(독자가 한 번에 이해하려는 단위)를 문서 장르(가이드/인덱스/레퍼런스)나 추상 계층으로 쪼개면 길찾기 비용이 길이 부담을 넘어 더 헷갈린다.
- **끝이 흐지부지** → 요약 또는 체크리스트로 닫는다.
- **깊은 디테일 복붙** → 정본으로 링크하고 본문엔 핵심만.

### 교육 안티패턴 (하지 말 것)

- 전제·큰 그림 없이 바로 디테일 / 예시 없는 추상 규칙 / "그냥 이렇게 해"(왜 없음) / 용어를 안 풀고 쏟기 / 한 문서에 너무 많은 주제 / **한 주제를 여러 문서로 과분할(한 번에 이해할 단위가 N개 파일로 흩어짐)** / 같은 내용을 여러 문서에 복붙.

> 레퍼런스(4-아키텍처/3-기법)는 terse 가 맞지만, **큰 그림·"왜" 콜아웃** 은 거기에도 도움이 된다 — 입문 전용은 예시·용어풀이·직접해보기 쪽이다.

## 이 가이드를 스스로 갱신한다 (self-improving)

이 파일은 고정 규칙이 아니라 **문서를 쓰며 자라는 살아있는 가이드**다. 새 문서를 작성/개정할 때마다, 같은 변경 안에서 아래를 함께 한다:

- **반복 구조 승격** — 여러 문서에 같은 골격·표현·콜아웃이 반복되면 → 여기 규칙(또는 "종류별 변형" 표)으로 올린다.
- **더 나은 패턴 채택** — 기존 규칙보다 나은 방식을 쓰게 됐으면 → 규칙을 고치고, 정본 예시를 그 문서로 교체한다.
- **새 종류/폴더** — 기존 골격에 안 맞는 문서 유형을 만들면 → 폴더 표 + "종류별 변형" 표 + README 에 한 줄씩 추가한다.
- **죽은 규칙 삭제** — 어느 문서도 안 따르는 규칙은 지운다.

원칙: 규칙은 실제 문서에서 **귀납**한다. 안 쓸 규칙을 예방적으로 넣지 않는다(speculative 금지). 보고서가 아니라 다음 작성자를 위한 전방 지시로 쓴다.

## 새 문서 추가 시

1. 폴더 선택(위) → `<folder>/<주제>.md` 생성, 종류에 맞는 골격으로 채움.
2. [README.md](README.md) 해당 섹션 표에 한 줄 (링크 · 요약 · 범위). 없는 폴더면 폴더 표에도 한 줄.
3. 새 골격/관례가 생겼으면 위 "이 가이드를 스스로 갱신한다" 에 따라 이 파일도 같이 고친다.
